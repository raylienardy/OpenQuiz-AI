# Mock Provider

## Overview

The Mock provider is a built‑in AI client that returns **fixed, deterministic responses** without calling any external API. It is ideal for:

- Frontend development when you don’t want to wait for real AI responses.
- Offline work (no internet required).
- Automated testing (predictable output).
- CI/CD pipelines that cannot access external services.

## Configuration

Set a single variable in `backend/.env`:

```env
AI_PROVIDER=mock
```

No API keys, no internet, no quotas.

## How It Works

`MockClient` implements the same `BaseAIClient` interface as Gemini and Groq. When `AIService` calls `generate()`, the mock client immediately returns a hard‑coded JSON response containing sample quiz questions, mimicking a real provider.

## Example Response

Calling `GET /ai/test` with Mock returns:

```json
{
  "success": true,
  "provider": "mock",
  "model": "mock-model",
  "response": "{\n  \"questions\": [\n    {\n      \"question\": \"This is a mock question.\",\n      \"choices\": [\"Option A\", \"Option B\", \"Option C\", \"Option D\"],\n      \"answer\": \"A\",\n      \"explanation\": \"This answer is generated by Mock AI.\"\n    }\n  ]\n}"
}
```

The exact content is fixed and will not change between calls.

## Extending the Mock Provider

In the future, the Mock provider can be enhanced to simulate real‑world conditions (optional):

- **Delay** – add artificial latency.
- **Random failures** – test error handling paths.
- **Invalid responses** – test robustness of parsers.

These extensions would be activated via additional environment variables (e.g., `MOCK_DELAY_MS=500`, `MOCK_FAIL_RATE=0.1`).

## When to Use Mock

- You’re building the frontend and need a consistent data source.
- You’re writing unit/integration tests for components that consume AI responses.
- You’ve exhausted your free tier on other providers and want to keep coding.

The Mock provider ensures **zero external dependencies** during development and testing.
